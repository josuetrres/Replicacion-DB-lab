🐘 Asistente de Failover PostgreSQL

        
Guía paso a paso para simular una falla en el master y promover una réplica.

      


      

        
⚠

        

          Antes de comenzar: Asegúrate de que el stack esté corriendo y que en el Tablero todos los nodos aparezcan como conectados.
        

      


      
      

        
Progreso del Failover

        

          

        

        

          
1

          
2

          
3

          
4

        

      


      


        
        

          

            
Paso 1

            
Verificar que la Replicación está Sana

          

          

            
Confirma que ambas réplicas están recibiendo el flujo WAL del master. La API consultará pg_stat_replication.

            

              
            

            

            

              Se espera: Dos filas con state = streaming
            

          

          

            
          

        


        
        

          

            
Paso 2

            
Simular la Caída del Master

          

          

            

              💀
              Acción destructiva: Esto detiene el contenedor pg-master vía la API de Docker. Las escrituras fallarán hasta promover una réplica.
            

            

              
            

            

            
Verifica que las réplicas sigan corriendo pero en modo lectura (recovery):

            

              
            

            

              Se espera: in_recovery = true (sigue siendo standby)
            

            

          

          

            
            
          

        


        
        

          

            
Paso 3

            
Promover la Réplica 1 a Nuevo Master

          

          

            
Promueve pg-replica1 a primario mediante la API. Esto sale del modo de recuperación, habilitando lectura y escritura, y finalizando el streaming del WAL antiguo.

            

              
            

            

            

              Se espera: Nodo promovido exitosamente.
            

            
Prueba hacer una escritura para confirmar que permite escribir:

            

              
            

            

          

          

            
            
          

        


        
        

          

            
Paso 4

            
Observar Réplica 2 — Estado Huérfano

          

          

            

              ⚠
              La Réplica 2 ha quedado huérfana. Sigue configurada para recibir WAL de pg-master (detenido) y no puede unirse automáticamente a pg-replica1.
            

            
Observa el error del WAL receiver en la Réplica 2:

            

              
            

            

            
Para reconectar la Réplica 2 al nuevo master, debes limpiar sus datos y volverla a levantar:

            

              
docker compose stop pg-replica2
docker run --rm -v lab_replica_pg_replica2_data:/data alpine sh -c "rm -rf /data/*"
# Editar docker-compose.yml: poner PG_MASTER_HOST: pg-replica1 para pg-replica2
docker compose up -d pg-replica2

              
            


            

              
📚 Concepto Clave

              
En entornos de producción, herramientas como Patroni, repmgr, o pg_auto_failover automatizan todo este proceso, incluyendo reconexión de réplicas huérfanas, elecciones de primario y cambios de DNS/IP.

            


            

              
🔄 Restaurar el Laboratorio

              

                
docker compose down -v
docker compose up -d --build

                
              

            

          

          

            
            
          

        


      


      
      

        
Resumen del Failover en PostgreSQL

        

          
PasoAcciónResultado

          
            
1API: /pg/statusAmbas réplicas streaming WAL

            
2API: Detener pg-masterMaster caído, réplicas en solo-lectura

            
3API: Promover pg-replica1Réplica 1 pasa a ser el nuevo primario

            
4Observar pg-replica2⚠ Huérfana — no se reconecta automáticamente
